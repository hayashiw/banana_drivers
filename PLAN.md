**`dof_scale` is a coordinate rescaling, NOT a step bound (2026-04-09).** Investigation of SIMSOPT's `augmented_lagrangian.py` confirmed: `dof_scale` maps DOFs to a rescaled space ($y = x / \text{scale}$). Each inner L-BFGS-B iteration is smaller in physical space, but the inner loop runs many iterations (up to `maxfun` function evals) and their displacements accumulate freely. Run 51253798 showed $\|\Delta x\|=0.102$ with `dof_scale=0.005` ŌĆö the raw L-BFGS-B direction had magnitude ~20, and 50 inner iterations accumulated 0.102 physical displacement. The only hard cap on total displacement per outer iteration is `maxfun`.

**BoozerLS `constraint_weight` bump 1.0 ŌåÆ 1.0e+3 (2026-04-09).** `boozer.constraint_weight` controls the weight on BoozerLS's surface-label penalty term relative to the Boozer residual in the least-squares problem. The larger the weight, the more firmly BoozerLS anchors the iota iterate to the requested label (via $(\iota - \iota_\text{label})^2$-type terms in the residual) rather than letting it drift to whatever minimizes the Boozer residual alone. The prior value `1.0` allowed drift into the spurious iota=0 basin on any surface with weak helical coupling to the coil field (see `diag_iota_basin.py` results + helical amplitude hypothesis above). `1.0e+3` matches SIMSOPT example conventions for stiffer anchoring. This is an **independent mitigation** from the coil capability problem ŌĆö it does NOT manufacture a basin that doesn't exist (Poincare 51287878 showed the coils can't support $\iota=0.15$ regardless of BoozerLS settings), but it should reduce the risk of BoozerLS wandering into degenerate iota=0 basins when the coil field DOES support the target, making future cold-start pipeline runs more reliable once a coil-capable $\iota_\text{max}$ is chosen. Applies to singlestage (`03_singlestage_driver.py`) and the standalone `utils/init_boozersurface.py` CLI which both read the same config key.

**Poincare `R_axis` starting-point patch (2026-04-09).** `poincare_tracing.py:make_start_points` previously estimated the magnetic axis $R$ as the arithmetic mean of inboard/outboard midplane points ($z\approx 0$) on the $\phi=0$ cross section, which assumes $R_\text{axis}$ is halfway between inboard and outboard ŌĆö only true for an up-down-symmetric ellipse with zero triangularity. For positive triangularity the true axis sits inboard of the midplane mean (Shafranov-like shift from shape alone). Switched to a weighted heuristic `r_axis = 0.25 * (3*r_inboard + r_outboard)` ŌĆö shift the starting-sweep origin 3:1 inboard ŌĆö so the starting R sweep covers the expected axis region instead of landing on the outboard side of it. Midplane mean is kept in the log for comparison. The quadpoint-centroid alternative (`cs0_r.mean()`) is present in the source as a commented reference for future experiments; the 3:1 heuristic was chosen because it gives a more conservative inward bias that handles circular-core surfaces (where the centroid equals the midplane mean) without regressing. Rationale: on surfaces whose $\iota$ is too small to support nesting much beyond the axis, getting the starting R close to the real axis matters.

Lagrange multipliers frozen at `[-8.35e6, 0, 0, -2.17]` from iter 15 onwards; ╬╝ exploded to 1e60 while state was frozen at 9 sig figs across 80 iters. Root cause: **SquaredFlux was the only non-self-clipping term in the constraint list**. Length/CC/curv are wrapped in QuadraticPenalty or are self-clipping threshold penalties (c=0 when below limit), so ALM correctly considers them "satisfied" when inactive. SquaredFlux is a smooth unbounded-below objective that never hits 0, so ALM always considers it active, drives it unboundedly via ╬╝-ramp, and the resulting descent direction pulls coils into shapes that violate every geometric constraint. The Lagrange-multiplier update rule then fails for the geometric constraints because they're in the violated-but-no-update regime.

**Fix applied (2026-04-08, updated 2026-04-09):** SquaredFlux uses `definition="normalized"` (closes the "shrink the field" loophole) and has a native `threshold` parameter (ported from PedroGil's `simsopt_alm_temp` fork into `fluxobjective.py`): `J()=0` and `dJ()` returns zero `B_vjp` once `sq_flux < threshold`. This gives a clean inactive state (`c=0`, `Ōłćc=0`) for ALM. Earlier (2026-04-08) this was done via `QuadraticPenalty(Jsqf, sqf_target, "max")` wrapping, but that created a ghost $\lambda/\mu$ ratchet (see 51251125, 51251674). The threshold is `1e-15` ŌĆö a noise floor, not a convergence target (run 51252548 showed that `1e-3` causes catastrophic CC collapse). Banana current pinned at 16 kA (`mode=fixed`).

**Stage 1 phiedge cache bug (2026-04-08).** Warm-start stage 1 was producing wouts with $|B|\sim 12\text{ T}$ (seed wout has ${\sim}1\text{ T}$). Root cause: `Vmec(mpi=mpi)` loads `input.default` at construction, which sets `phiedge=1.0`. The Optimizable `__init__` calls `self.get_dofs()` once and caches `[1.0, 0.0, 1.0]` in `local_full_x`. **Stage 2 current collapse to 0 kA (job 51246996, 2026-04-08).** First stage 2 ALM run after adding `current` as a 5th constraint (`QuadraticPenalty(|I|, 20\text{ kA}, "max")`) as a warm-start-friendly soft cap. The ALM descent correctly follows the gradient to $I=0$. Once there, the banana B-field is $I \cdot B_\text{unit}(\text{shape})$, so every shape gradient carries a factor of $I$ and vanishes. $\partial J/\partial I$ at $I=0$ is not exactly zero analytically, but `numpy.sign(0)=0` in `CurrentPenaltyWrapper.dJ()` plus the already-zero shape gradients pin the system at a degenerate stationary point. The trivial solution ($I=0$, all constraints 0) is lower-$J$ than anything the optimizer can reach by perturbing shapes.

## Augmented Lagrangian Method (ALM) ŌĆö Stage 2 port complete

**Motivation.** Stage 2 job 51219280 (2026-04-08) failed because L-BFGS-B's initial descent direction from a fresh `boozersurface.init.json` walked into the `LpCurvCurv` penalty cliff. The `p=4` curvature penalty with threshold 20 mŌü╗┬╣ creates a sharp nonlinear wall; once the optimizer is on the wall at iter 74, the recommended step blew J up by 14 orders of magnitude (9.6e-4 ŌåÆ 5.3e+10), line search could not recover, and the run terminated at machine precision with large gradient. This is starting-point-sensitive and not robust enough for Pareto scans.

**Dead end ŌĆö two-phase warmup (Option B).** Before ALM, a temporary two-phase driver was tried: Phase 1 minimized `SquaredFlux` alone (no penalty walls), Phase 2 enabled the full weighted objective. Phase 1 let the coils grow 11├Ś unconstrained (length 0.89 ŌåÆ 9.87 m, CC collapse to 4.2 mm, curvature 2197 mŌü╗┬╣); Phase 2 could not recover. `SquaredFlux` alone is pathologically unbounded ŌĆö this approach is dead. See job 51221431.

**ALM approach.** Set `f=None` and place **all** objectives ŌĆö including SquaredFlux ŌĆö in the constraint list. ALM treats each as an equality-style residual, ramps per-constraint penalty weights $\mu_i$ in an outer loop, and updates Lagrange multipliers $\lambda_i$ as constraints become satisfied. The effective weight on each constraint's gradient is $(\mu_i c_i - \lambda_i)$, analogous to $w_i$ in a traditional weighted objective. Inner loop is L-BFGS-B on a smooth augmented Lagrangian ŌĆö no penalty cliffs.
